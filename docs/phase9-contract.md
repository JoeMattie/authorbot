# Phase 9 implementation contract — documentation and the project site

Additive to Phase 0–8 contracts. Follows Phase 6 because the site documents
the wizard, the "New chapter" button, and settings — none of which can be
screenshotted before they exist.

**Goal:** a novelist who has never heard of Authorbot can land on
`authorbot.joemattie.com`, understand within thirty seconds whether it is for
them, and be running the wizard a minute later. Separately: the repository
stops greeting newcomers with implementation contracts.

## 1. The problem

Everything written so far is engineering documentation: seven phase contracts,
twenty-plus ADRs, a design document written for an implementer. That is
correct and should stay. But it is *all* we have, and the README leads with
architecture and phase status — so an author arriving from a link sees a
monorepo layout and a roadmap table rather than "this helps you write a serial
novel with other people."

Two audiences with opposite needs are currently sharing one surface.

## 2. Audience separation

| Audience | Wants | Lives in |
|---|---|---|
| **Authors** | what it does, how to start, how to do a task | `authorbot.joemattie.com` |
| **Contributors/operators** | why it works this way, invariants, contracts | the repository |

Contracts, ADRs, and the design document are **not published to the site**.
They stay in the repository, linked once from a "How it's built" page for the
curious. Publishing them would bury the four pages an author actually needs.

## 3. Repository README

Rewritten for a first-time visitor, in this order: what Authorbot is (two
sentences), who it is for, a screenshot, the one-line quickstart, what it
costs (a GitHub account, a Cloudflare account, free tiers), and *then* links —
site, getting started, how it works. Roadmap, monorepo layout, and phase
contracts move below the fold or into `CONTRIBUTING.md`.

The existing `docs/how-it-works.md` and `docs/getting-started.md` are rewritten
for authors and become the site's source (§5), leaving the repository copies
as the canonical text rather than a duplicate.

## 4. The site

Static, deployed to `authorbot.joemattie.com` on Cloudflare — the same shape
as a book site, so we operate one kind of thing.

**Pages:**

1. **Landing** — what it is, the core promise ("Authorbot manages authorship.
   It does not perform authorship."), a screenshot of the annotation gutter,
   the quickstart command, and honest scope: what it does *not* do (no LLM, no
   hosting, no lock-in).
2. **Quickstart** — the wizard, start to finish, with what each stage asks and
   what you get. The page a link should point at.
3. **How it works** — the three planes and the lifecycle of one change, in
   author language. Adapted from `docs/how-it-works.md`, with its ASCII
   diagrams redrawn as SVG for the web while the text stays canonical in the
   repository.
4. **Guides** — task-shaped, one page each: write and publish a chapter;
   invite a collaborator; tune governance; work with agents; upgrade;
   troubleshoot.
5. **Reference** — CLI commands, the book repository format, and the API,
   **generated from source** (§6) rather than retyped.
6. **How it's built** — one page for the engineering-curious, linking the
   design document, contracts, and ADRs.

**Not in scope:** a blog, a changelog beyond release notes, versioned doc
trees (single current version until there is a v2), search (the site is small
enough to navigate), and analytics — consistent with a project that collects
nothing.

## 5. Screenshots that cannot rot

Screenshot drift is the standard failure of documentation sites, and manual
recapture never happens. So: **screenshots are generated by the existing
Playwright suite**, not taken by hand.

- A `docs-screenshots` project in the Playwright config drives the real UI
  against the dev API with seeded fixture content, capturing each documented
  view at a fixed viewport in both colour schemes.
- Output is committed, and CI fails when a regenerated screenshot differs
  beyond a small threshold — the same signal a broken test gives.
- Every screenshot on the site is produced this way. A screenshot that cannot
  be generated from a running system does not go on the site.

## 6. Anti-drift for text and code

- **Code samples are executed.** Every shell block in quickstart and the
  guides is extracted and run against a scratch book repository in CI, or is
  explicitly marked non-executable with a reason.
- **CLI reference is generated** from the commands' own `--help` output.
- **API reference is generated** from `openapi/openapi.yaml`, which Phase 3
  already keeps synchronised with the router by test.
- **Version claims are generated** from the published npm version, never
  typed.
- The site build runs in CI and **fails on a broken internal link**.

## 7. Deployment

A Cloudflare Worker serving static assets, deployed from this repository on
push to `main`, with `authorbot.joemattie.com` as a custom domain. No API, no
database, no client JavaScript beyond what a diagram or copy button needs.
Accessibility matches the standard applied to book sites (design §16.6):
keyboard navigable, sufficient contrast, dark mode, readable without
JavaScript.

## 8. Optional: a live demo

A read-only demonstration book with seeded annotations, votes, and a work
queue, so a visitor can see the collaboration UI without installing anything.
**Read-only**: no sign-in, no writes, no moderation burden, no abuse surface.
Deferred if it competes with Phase 7 hardening for attention — a good
screenshot conveys most of the value at a fraction of the cost.

## 9. Exit criteria

1. A reader unfamiliar with the project can state, after thirty seconds on the
   landing page, what Authorbot does and what it does not do.
2. Quickstart takes an author from zero to a published book without leaving
   the page or reading the repository.
3. Every screenshot is generated by the Playwright suite; regenerating them
   produces no diff.
4. Every executable code sample runs green in CI; the site build fails on a
   broken internal link.
5. CLI and API reference pages are generated, not hand-written.
6. The README leads with the author's question, not the architecture.
7. No contract, ADR, or design-document content is republished to the site.
8. `authorbot.joemattie.com` deploys from `main` and is keyboard navigable,
   dark-mode correct, and readable without JavaScript.

## 10. Sequencing note

**This phase runs last, including the README rewrite.** An earlier draft
suggested landing the README as soon as the wizard existed. That was wrong:
it would need revising when npm distribution replaces the clone-and-build
instructions (Phase 6 / ADR-0022), again when the collaborator skill adds an
install path for agent users (Phase 8), and again for the site's own links.
Writing it once, against a settled surface, costs less and reads better.

Accepted consequence: the README stays engineer-facing until then. That is
honest — until Phase 6 the tool is not installable by a novelist, so the only
audience who can act on it today is the one it currently addresses.
